How Finance AI Radar is generated, refreshed daily, deployed, and self-learns

This document maps the full operating system behind Finance AI Radar: the Next.js static-site build, source-data model, editorial gates, ingestion scripts, local learning files, Hermes cron jobs, GitHub Pages deployment helper, and the current self-learning loop. It is intentionally detailed enough for a future agent or human maintainer to modify the site without rediscovering the pipeline.

Inspected project path: /Users/nyp/Documents/my_life/scratch my itch to learn and prosper/finance-ai-radar. Public URL: https://neoyipeng2018.github.io/finance-ai-radar/. Deployment helper: /Users/nyp/.hermes/scripts/deploy_finance_ai_radar_pages.py.

1. Executive summary

31reviewed source items in data/sourceItems.ts
16source categories represented, including GitHub, arXiv, datasets, jobs, Hugging Face models and papers
2daily Hermes cron agents: 03:00 SGT review and 05:00 SGT update/deploy
$0current hosting cost via GitHub Pages static export

Finance AI Radar is currently a static exported Next.js application. The rendered pages are produced at build time from TypeScript and TSV/JSON files in the repo, then force-pushed to the gh-pages branch of git@github.com:neoyipeng2018/finance-ai-radar.git. The site is refreshed by a two-stage Hermes Agent loop: a 03:00 SGT review agent produces/updates implementation guidance, and a 05:00 SGT production agent makes one coherent update, runs verification, builds with the GitHub Pages base path, deploys, and checks the live URL.

Most important intricacy: the repository has both a source tree inside the broader my_life repo and a deployment helper that syncs the source into the standalone GitHub repository neoyipeng2018/finance-ai-radar. The local source is not itself a standalone Git repo; its immediate git status resolves to my_life. Deployment therefore depends on the helper cloning and syncing the standalone public repo before publishing out/.

6. How pages are generated

The site uses Next.js App Router pages, but because output: 'export' is enabled, these pages become static HTML during next build. There is no database at request time.

RouteSource inputsGeneration behavior
/data/sourceItems.ts, data/review_queue.tsvImports source items, computes featured items, source counts, themes, review queue summary, editorial metrics, hiring signals, monetization blocks, and newsletter box.
/datasets/sourceItems filtered by datasetFieldsShows reviewed public finance/NLP datasets and coverage chips such as modalities and NLP use cases.
/huggingface/sourceItems filtered to Hugging Face datasets/models/papersShows models and papers with coverage summaries for model tasks and finance domains.
/jobs/sourceItems filtered to sourceType === 'job'Frames AI + finance jobs as demand-side intelligence and lists skill/finance-domain signals.
/review/data/review_queue.tsvReads candidates at build time, summarizes status counts and stale candidates, and displays unpromoted candidates for editorial review.
/sponsor/Static component contentSupports sponsor/media-kit monetization surface. Paid/sponsored content must remain labeled and must not silently influence editorial ranking.

Ranking and search details

  • getFeaturedItems(items, limit) filters to reviewed/published, sorts by descending score, then descending readingTimeMinutes, then title.
  • searchItems() filters by source type and theme, then searches title, publisher, source type, summary, rationale, use case, technique, caveat, themes, audience, and optional job/dataset/Hugging Face metadata fields.
  • getEditorialMetrics() computes reviewed coverage, rights-safe item count, high-evidence count, source diversity, coverage gaps, queue size, stale candidates, dataset coverage, jobs coverage, and Hugging Face coverage.

The page-generation pipeline is deterministic: modify source files, run build, and the output changes. There is no remote CMS, database, server component mutation, or live ingestion at page-request time.

7. Daily refresh loop

The daily refresh is orchestrated by Hermes cron jobs rather than GitHub Actions. The scheduler currently has two Finance AI Radar jobs, both enabled, both running in the project workdir, both using the test-driven-development and systematic-debugging skills, and both limited to terminal/file/web toolsets.

Produce a concise implementation brief for the 05:00 agent: highest-value improvement, rationale, acceptance tests, rollback note, and rights/license caveats. 05:00 job required process

  1. Read AGENTS.md and the latest data/daily_reflections.tsv entry/review notes.
  2. Make one coherent, evidence-backed update. Allowed categories include reviewed sources, public-domain/NLP datasets, Hugging Face finance models/papers, AI + finance jobs intelligence, sponsor/media-kit surface, quality console, or production reliability.
  3. Respect the editorial rule: every published item needs a source link, finance use case, why it matters, caveat, license/rights note, and review status.
  4. Run npm test, npm run typecheck, and NEXT_PUBLIC_BASE_PATH=/finance-ai-radar npm run build.
  5. Deploy with python3 /Users/nyp/.hermes/scripts/deploy_finance_ai_radar_pages.py.

Subtle timing design: the 03:00 agent is a review/planning/reflection pass. The 05:00 agent is the production updater/deployer. This reduces the chance that a production agent chooses an impulsive change without prior context, while still allowing it to act independently if the review output is minimal.

8. Deployment mechanics

Deployment is centralized in /Users/nyp/.hermes/scripts/deploy_finance_ai_radar_pages.py. The helper performs both source synchronization and static-site publishing.

Deployment helper constants

ROOT = /Users/nyp/Documents/my_life/scratch my itch to learn and prosper/finance-ai-radar

Deployment helper sequence

  1. Run npm install --include=dev in ROOT. This matters because this environment has npm configured to omit dev dependencies unless explicitly included.
  2. Run npm test.
  3. Run npm run typecheck.
  4. Run npm run build with NEXT_PUBLIC_BASE_PATH=/finance-ai-radar in the environment.
  5. Delete and reclone git@github.com:neoyipeng2018/finance-ai-radar.git into /tmp/finance-ai-radar-public-sync.
  6. Remove all files from that cloned worktree except .git.
  7. Copy current source files from ROOT into the public worktree, excluding node_modules, .next, out, and .git.
  8. Commit source changes with chore: sync daily finance ai radar source if there are changes, then push to origin main.
  9. Touch out/.nojekyll to stop GitHub Pages from running Jekyll processing.
  10. Remove any stale out/.git, initialize out/ as a new gh-pages repo, add remote, commit all static output with deploy: daily finance ai radar static site, then force-push origin gh-pages.
  11. Print Finance AI Radar deployed: https://neoyipeng2018.github.io/finance-ai-radar/ (source_changed=...).

Why the base path is non-negotiable

The public site is served from a project page path, not the domain root. Therefore the correct production build must use NEXT_PUBLIC_BASE_PATH=/finance-ai-radar. The Next config maps this to both basePath and assetPrefix. Building without that environment variable may make local export look fine but will break asset and route references when hosted under /finance-ai-radar/.

Force-push warning: the helper always reinitializes out/ and force-pushes gh-pages. This is normal for static deploys but means gh-pages history is disposable and must not be used as the source of truth.

9. Self-learning and feedback system

The self-learning loop is not an online model that trains weights. It is an agentic editorial/product learning loop: collect or summarize signals, form a hypothesis, make a small production change, verify it, record the rationale/rollback note, and let future agents read that ledger before acting.

Learning memory surfaces

SurfaceWhat it storesHow future updates use it
AGENTS.mdMission, editorial rule, current learning, ranking heuristics, dataset/Hugging Face/jobs priorities, daily loop rules, experiment backlog.Both daily cron agents are prompted to read this first. It acts as the durable operating policy.
data/daily_reflections.tsvHypotheses, decisions, files changed, verification results, rollback notes.The 05:00 agent reads the latest reflection before deciding the next coherent update. This creates a human-auditable experiment ledger.
data/daily_metrics.jsonSummarized user behavior such as sessions, source clicks, top items/source types/themes/jobs, average scroll depth, high-intent paths.The 03:00 agent reviews it to identify what users engage with, then can suggest changes or update learning notes.
data/review_queue.tsvCandidate sources with license guesses and reviewer notes.Agents can mine candidates but must not publish them until they have full editorial fields and reviewed/published status.
data/source_ledger.tsvAudit trail for published/reviewed source rationale and rights posture.Keeps source decisions explainable outside of TypeScript code.
Hermes cron promptsStanding instructions about review/update cadence, verification, deployment, rights, and budget.Keep daily behavior aligned without needing a human every morning.

Self-learning update pattern

  1. Measure or inspect behavior/content gaps: review daily_metrics.json, source library coverage, queue candidates, public site UX, and ledgers.
  2. Form a concrete hypothesis, e.g. “Hugging Face models and papers deserve first-class coverage because they connect reusable assets, papers, datasets, and community attention.”
  3. Implement one coherent change: new page, new source type, new reviewed items, improved metrics surface, production reliability fix, etc.
  4. Update typed source data and human ledgers together when publishing content.
  5. Run tests/typecheck/build.
  6. Record the hypothesis, decision, changed files, verification, and rollback note in daily_reflections.tsv.
  7. Deploy and verify the public URL.

The inspected daily_reflections.tsv currently records two learning cycles: initial loop scaffolding and addition of Hugging Face model/paper coverage. AGENTS.md also notes that seeded metrics are only for system validation and warns future agents to wait for real click, dwell, filter, dataset, Hugging Face, job, and sponsor events before changing ranking heuristics.

Candidate-to-publication path

  1. A script or human adds a row to review_queue.tsv with status: candidate or triaged.
  2. The review page displays the queue and status summary at build time.
  3. An agent/human reviews the original source, license, rights, relevance, caveats, and finance use case.
  4. If publication-worthy, the item is manually promoted to a full ContentItem in data/sourceItems.ts with complete required fields and optional metadata objects.
  5. The source is also recorded in data/source_ledger.tsv for human auditability.
  6. Tests/typecheck/build verify the new type/source/coverage behavior.

11. Analytics reality and limitations

The codebase contains a first-party analytics system, but the production deployment is static GitHub Pages. That creates an important mismatch.

What the code supports

  • AnalyticsTracker runs client-side on every page via app/layout.tsx.
  • It sends page_view on mount and session_end with dwell time and scroll depth on unload.
  • TrackedSourceLink sends source_click or job_click.
  • sendAnalyticsEvent() only sends if NEXT_PUBLIC_ANALYTICS_ENDPOINT is set.
  • app/api/events/route.ts can validate and append events to data/analytics_events.tsv in a server-capable Next runtime.
  • summarize_metrics.py converts TSV events into daily_metrics.json.

What GitHub Pages changes

Static export limitation: GitHub Pages cannot run app/api/events/route.ts. With output: 'export', the public site is static. Therefore live first-party analytics will not append to local TSV through that route unless events are sent to an external endpoint or a different hosting/runtime is introduced.

At inspection time, data/daily_metrics.json contains a small seeded/system-validation snapshot: 2 sessions, 1 source click on dataset-sec-edgar, top source type dataset, top theme Public-domain finance text, one top job entry, average scroll depth 40, and no high-intent paths. AGENTS.md explicitly warns not to overfit ranking heuristics to these seeded metrics.

Implication for self-learning

Until a real analytics endpoint exists, the strongest learning signals are curated source reviews, ledgers, public-site inspection, review queue quality, and human/agent reflection. The analytics schema is valuable scaffolding but not yet a complete production feedback loop on GitHub Pages.

12. Verification gates

The repo expects concrete verification before deployment. The standard local gates are:

npm test
npm run typecheck
NEXT_PUBLIC_BASE_PATH=/finance-ai-radar npm run build

The deployment helper also runs:

npm install --include=dev
npm test
npm run typecheck
npm run build   # with NEXT_PUBLIC_BASE_PATH=/finance-ai-radar in env

Tests cover the following areas:

  • Library logic: source counts, theme briefs, featured sorting, search behavior, editorial metrics.
  • Review queue: TSV parsing, duplicate candidate detection, publishable status, stale-candidate summary.
  • Analytics: event schema validation, click/theme/job/dwell/scroll summaries.
  • Datasets/jobs/Hugging Face: minimum source coverage, metadata fields, labels, searchability.
  • Operating files: required scripts and ledgers exist.
Deployment verification requirement: the 05:00 cron prompt requires an HTTP fetch of the live URL after deployment and an exact report of the result. If deployment fails, the agent must leave a rollback note and must not claim success.

14. Operator playbook

Make a normal content/product update

  1. Read AGENTS.md, latest daily_reflections.tsv, and relevant candidates/ledgers.
  2. Choose one coherent change. Avoid mixing unrelated content, design, analytics, and deployment changes in one cycle.
  3. If publishing a source, ensure complete ContentItem fields: source link, finance use case, why it matters, caveat, license/rights note, review status.
  4. Update sourceItems.ts and source_ledger.tsv together. If the change is learning-related, update daily_reflections.tsv.
  5. Add/update tests if new source types, fields, metrics, or rendering logic are introduced.
  6. Run npm test, npm run typecheck, and NEXT_PUBLIC_BASE_PATH=/finance-ai-radar npm run build.
  7. Deploy with python3 /Users/nyp/.hermes/scripts/deploy_finance_ai_radar_pages.py.
  8. Fetch https://neoyipeng2018.github.io/finance-ai-radar/ and verify HTTP success and visible expected content.

Promote a queued item

  1. Inspect original source URL and rights/license terms.
  2. Determine source type, evidence quality, maturity, themes, audience, score, and reading time.
  3. Write original summary/rationale/caveat; do not copy proprietary or lengthy source text.
  4. Add optional metadata object when applicable: datasetFields, jobFields, huggingFaceModelFields, or huggingFacePaperFields.
  5. Set reviewStatus: 'reviewed' only after all mandatory fields are complete.
  6. Record the item in source_ledger.tsv.

Improve the self-learning system

  • First priority: connect real analytics. Since GitHub Pages cannot receive /api/events, add a zero/low-cost external endpoint or change hosting for analytics only.
  • Second priority: make daily metrics generation part of the cron process if real events arrive outside the repo.
  • Third priority: keep the reflection ledger unified and human-auditable. Do not split hypotheses and outcomes into disconnected files.
  • Fourth priority: preserve rights safety. The site’s moat is signal and auditability, not scraping proprietary finance content.
